- **新 Atlas 暂时占用现有 Atlas 入口位置**（Capture 顶部的 Atlas 按钮）。旧 5 分区视图代码移到 `views/_legacy/AtlasView.vue` 保留备查，不挂路由

### 状态机（v0.1 全部包含）

- Global Overview（球外远观）
- Region Zoom（推近到某主线扇区）
- Node Focus（聚焦单节点）
- Relation Reveal（聚焦上叠加关联线）

四态都做。`Esc` 单步回退，右上角文字面包屑可点祖先跳。

### 数据（v0.1）

- **完全使用** `frontend/public/mock/cosmos.json` 静态数据
- **完全不接** 任何后端 API（任何 fetch /memories /timeline /tasks 都禁止）
- **不动** 任何数据库表
- 关联数据全部用 mock 预定义，**不做** AI 拟合

### 不做的特性（v0.1 明确禁止）

- focus_session 录制（但**预留事件总线 hook**）
- reflection_prompt 反思提示
- 底部 AI 进入引导文案（mock 数据生成的引导假）
- evidence 抽屉详情面板
- 关联 accept/reject 用户审核按钮
- 球面纹理 / 星空背景 / 粒子特效
- 三模式入口改造

### 技术栈（v0.1）

- 保留 Vue 3 + TS + Pinia + Vite
- `three` + `@types/three` 新增依赖，**懒加载**（独立 chunk，不拖累 Capture 启动）
- **不用** TresJS / GSAP / react-three-fiber 等包装层
- 摄像机插值用自己写的 60 行 tween util（requestAnimationFrame + cubic-ease）

### 长期方向（v0.2+，本 RFC §1~§10 描述）

球外/球内混合穿入、focus_session 路径录制、AI 关联拟合作业、反思提示持久化、三模式 tab、近况重命名等——全部在 v0.2+，**不在 v0.1 范围**。

## 0. 摘要

Axiom 当前的 Atlas（5 分区驾驶舱）本质是个数据看板，不符合"低频深度宏观思考"的产品定位。本 RFC 提出：

- 当前 Atlas → 改名 **"近况"**，作为二级入口保留
- 重做 **Atlas（宇宙）** 作为产品的第二个核心模式
- 视觉隐喻：球形放射树状结构 + 天文学美学
- 数据模型新增 4 张表（lifeline / tree_edge / association_edge / focus_session）
- v0.1 原型不接真实后端，只用 `mock/cosmos.json` 验证四个核心状态

## 1. 三模式边界

| | **Capture** | **Atlas** | **近况** |
|---|---|---|---|
| **目的** | 快速记录、找回 | 思考人生、复盘主线 | 看本周数据 |
| **频率** | 每天几十次 | 每周 1~3 次 | 每天 1~3 次 |
| **用户状态** | 碎片时间、心流不打断 | 有时间冷静思考 | 短暂关心当前状态 |
| **视觉调性** | 极简工具感、单栏 720px | 沉浸宇宙感、3D 全屏 | 数据密度、网格 1100px |
| **失败标准** | 输入太慢、AI 路由错 | 看完不知道自己往哪走 | 数字看不懂 |
| **可写操作** | 全部 | **只读思考** | 只读 |
| **入口** | 默认进入 | 顶部一级切换 | Capture 顶部二级图标 |

**关键边界**：
- Atlas **不允许**写入/编辑/归档/删除——任何写动作都引导用户 `Esc` 回 Capture
- Atlas 不显示数字 KPI——那是"近况"的事
- 近况不显示树/网络——那是 Atlas 的事

## 2. Atlas 的 4 个核心状态

Atlas 是一个**状态机**，不是 5 个面板。任何时刻处于以下 4 个状态之一。

### 2.1 Global Overview（球形树状总览）

| | |
|---|---|
| **屏幕中心** | 整个球体（球心略偏屏幕中心，留出右上面包屑空间） |
| **用户能做的** | 拖拽旋转 / 滚轮缩放 / 悬停看主线名 / 点击主线进 Region Zoom |
| **显示** | 根节点（"我"，最内层球壳 R₀） + 第一层主线（5~12 个，球壳 R₁） + 同心轨道环（alpha 0.04） + 底部 AI 一句话引导："你已经 12 天没动'减脂'——要看看吗？" |
| **隐藏** | 所有第 2 层以下节点、所有关联线、所有节点详情、所有操作按钮 |
| **退出** | 右上 `× 退出 Atlas` → 回 Capture。无"返回上一层"（这是顶层） |

### 2.2 Region Zoom（语义区域放大）

| | |
|---|---|
| **屏幕中心** | 被选中的主线及其扇区（如"健康"区） |
| **用户能做的** | 继续旋转/缩放（受限于该扇区附近）/ 悬停看项目名 / 点击二级节点进 Node Focus / 缩出回总览 |
| **显示** | 该主线本身 + 其下第二层节点（项目 / 主题，3~10 个）+ 该扇区轨道弧 + 其余主线**半淡**（alpha 0.15，仅作方位参考） |
| **隐藏** | 其他主线的子节点、所有第 3 层（叶子）、所有关联线 |
| **退出** | 滚轮缩出 / `Esc` / 点空白区域 → 回 Global Overview |

### 2.3 Node Focus（节点聚焦）

| | |
|---|---|
| **屏幕中心** | 被点击的单个节点（项目 / 知识点 / 决策 / 任务 / 目标）|
| **用户能做的** | 浏览节点元信息 / 按 `R` 切到 Relation Reveal / 点父节点回退一层 / `Esc` 回退 |
| **显示** | 节点本身放大 + 父链连线 + 直接子节点（若有）+ 右侧轻面板（创建时间、类型、来源、相关 task/memory 数）+ 底部 AI 反思提示（停留 >5s 出现）+ 右上面包屑 `健康 › 减脂 › 晨跑 5km` |
| **隐藏** | 树的其余大分支（fade 到 alpha 0.06，仅留方位感）、关联线 |
| **退出** | 面包屑任一祖先点击 / `Esc` → 回 Region Zoom（或祖先所在的最近状态） |

### 2.4 Relation Reveal（关联揭示）

| | |
|---|---|
| **屏幕中心** | Node Focus 的同一个节点 |
| **用户能做的** | 悬停关联线看 micro-tooltip / 点击关联线展开 evidence 抽屉 / 点击关联节点跳过去（变成新焦点）/ 再按 `R` 关闭关联 |
| **显示** | 当前节点 + 它的所有 `confidence ≥ 阈值（默认 0.7）` 关联边（粗细=置信度、颜色=类型）+ 关联到的对端节点（不管它们树归属在哪都拉出来显示）+ 右下角关联统计："共 12 条关联 · 8 已确认 · 4 待审核" |
| **隐藏** | 与当前节点**无关联**的所有节点褪到 alpha 0.05（几乎不可见）、树的其余轨道环 |
| **退出** | 再按 `R` / 点空白 → 关联线渐隐回 Node Focus |

### 2.5 状态机示意

```
                  Capture （Esc / × 退出）
                     ▲
                     │
              Global Overview
                  │   ▲
        点主线    │   │ Esc / 缩出
                  ▼   │
              Region Zoom
                  │   ▲
        点节点    │   │ Esc / 面包屑
                  ▼   │
              Node Focus  ◀──┐
                  │          │
              按 R│          │ 按 R / 点空白
                  ▼          │
              Relation Reveal─┘
```

切换都有 300~800ms 的平滑过渡。**没有跳变**。

## 3. 空间隐喻：球外 vs 球内

是否"根节点必须在球心"——重新讨论。

### 3.1 方案 A：用户在球外观察（天球仪式）

摄像机轨道围绕球体外侧，节点都在球面上，用户像看一颗有标签的星球。

**优点**：
- 所有节点可见性预期一致（不会被身后遮挡）
- 空间认知简单——像看一个 globe
- Global Overview 状态最自然，第一眼就 grok

**缺点**：
- 节点都在"远方"，聚焦时焦点不强
- Node Focus 时摄像机需要"穿透"球壳进入节点，有违和感
- "沉浸感"弱

**适合状态**：Global Overview、Region Zoom

### 3.2 方案 B：用户在球内观察（天文台仰望）

摄像机位于球心附近，节点环绕你，像在天文台圆顶下看星空。

**优点**：
- 沉浸感强，"个人认知宇宙"隐喻贴合
- 聚焦节点时摄像机在球内移动，过渡自然
- 远视场视角下背景星点感真实

**缺点**：
- 背面节点被身后遮挡，必须旋转才能看到——增加认知负担
- 多层球壳时，内层球会挡住外层节点
- 不适合"快速扫一眼整体结构"

**适合状态**：Node Focus、Relation Reveal

### 3.3 方案 C：混合（**我的主张**）

| 状态 | 视角位置 |
|---|---|
| Global Overview | **球外**（距离球 1.6 倍半径） |
| Region Zoom | 球外推近（距离球 1.1 倍半径，几乎贴壳） |
| Node Focus | **穿入球壳**，摄像机靠近节点（距节点 0.3 单位） |
| Relation Reveal | 同 Node Focus，但视场略放宽看周围关联线 |

状态切换时摄像机**穿过球壳**——这是从"看一颗行星"到"行星表面街景"的尺度切换，符合天文学多分辨率隐喻。

### 3.4 根节点是否在球心

**结论：根节点不放球心**。

理由：球心是抽象的"观察点"，不该放可点击对象。但根节点需要可见（"我"或"主题集合"是树的起点）。

方案：
- 根节点放在**最内层球壳 R₀**（很小，比如 R₀ = 0.2 * R₁）
- 球心是观察点，**Global Overview 时摄像机看向球心**
- 进 Region Zoom 时摄像机视线穿过球心略偏，看向被选主线扇区

## 4. 数据模型草案（仅字段，不写迁移）

### 4.1 `lifeline` — 人生主线树

```
id              uuid / int
name            string             // "健康"、"职业发展"
parent_id       lifeline.id?       // 自指，null = 顶层
order_index     int                // 同级排序
accent_hint     string?            // 视觉色 token 名（可选，AI 不能改）
status          'active' | 'paused' | 'archived'
created_at      timestamp
updated_at      timestamp
```

**关键判断**：lifeline 是**用户显式定义的**树。AI 可以建议新主线，但默认 `status='paused'`，需要用户激活。

### 4.2 `entity` — 通用挂载点（多态视图）

不是真表，是**(kind, ref_id) 元组**指向：

```
kind            'item' | 'task' | 'memory' | 'decision' | 'lifeline'
ref_id          目标表的主键
```

所有 tree_edge / association_edge 都用 entity 引用，避免每张表多对多。

### 4.3 `tree_edge` — 归属关系

```
id              uuid / int
from_kind       string             // 被归属的实体类型
from_id         int                // 被归属的实体 id
to_lifeline_id  lifeline.id        // 归属到哪条主线
created_at      timestamp
created_by      'user' | 'ai_accepted_v1'  // AI 提议被用户接受才能写
```

**v0.1 约束**：一个 entity **只能挂一条 tree_edge**（单归属，简化空间布局）。多归属留 v0.2 再议。

### 4.4 `association_edge` — AI 拟合的关联

```
id              uuid / int
from_kind       string
from_id         int
to_kind         string
to_id           int
relation_type   'co_occurrence' | 'causal' | 'tension' | 'derived_from'
confidence      float 0.0 ~ 1.0
evidence        json[]             // 见 §5
status          'pending' | 'accepted' | 'rejected'
created_by      string             // AI 模型版本，如 'qwen3_embed_v1'
created_at      timestamp
updated_at      timestamp          // 用户审核或 AI 重评时更新
```

### 4.5 `focus_session` — 探索路径录制

```
id              uuid
started_at      timestamp
ended_at        timestamp?
path            json[]             // [{entity_kind, entity_id, entered_at, dwell_ms}, ...]
mode            'free' | 'guided'  // 自由漫游 / AI 引导路径
```

**判断**：每次进 Atlas 默认开新 session；点击 ×退出 时关闭。**轻量自动**，不要求用户"开始一段思考"。

### 4.6 `reflection_prompt` — 驻留反思

```
id              uuid?              // 持久化 or 即时生成 — 见开放问题
entity_kind     string
entity_id       int
text            string             // AI 生成的反思问题
created_at      timestamp
shown_at        timestamp?
user_response   'engaged' | 'dismissed' | null
```

**初步判断**：v0.1 不持久化，即时生成；v0.2 看用户响应率再决定要不要存。

## 5. 关联边契约（必备字段细则）

### 5.1 必备字段

| 字段 | 说明 | 缺失行为 |
|---|---|---|
| `confidence` | 0.0~1.0 浮点 | 缺失 = 不渲染（必须有） |
| `evidence` | 至少 1 条 | 空数组 = 不渲染 |
| `relation_type` | 4 种枚举之一 | 缺失 = 当 `co_occurrence` |
| `status` | pending / accepted / rejected | 缺失 = pending |
| `created_by` | AI 模型版本或 'user' | 必须有 |
| `updated_at` | 任何状态变更必须刷 | 必须有 |

### 5.2 evidence 子结构

```
{
  "type": "shared_item" | "shared_keyword" | "temporal" | "explicit_link",
  "ref_id": int?,           // 如果是 shared_item，指向 item.id
  "excerpt": string,        // 给用户看的解释片段
  "weight": float           // 此条证据在 confidence 计算里的权重
}
```

每条关联**必须有出处**——这是 RFC 的硬规则，禁止 AI 玄学连线。

### 5.3 状态机

```
ai 拟合 → pending（默认）
  ├─ 用户点 ✓ accept → accepted（参与统计、不再褪色）
  ├─ 用户点 ✗ reject → rejected（永久不显示，但保留以便 AI 学习）
  └─ AI 重评 confidence 下降 < 阈值 → 自动 archived（保留记录）
```

用户**可以**手动创建一条 association_edge（`created_by='user'`，status='accepted'）。

### 5.4 渲染规则

- 默认只显示 `status IN ('accepted', 'pending') AND confidence >= 阈值`（阈值默认 0.7，用户可调）
- `pending` 关联线**虚线显示**，`accepted` 实线
- `rejected` 不渲染
- 数量超过 20 时按 confidence 降序裁剪到 20

## 6. 技术栈判断

### 6.1 现状

已扫描 `frontend/`：

- Vite 6.4 + Vue 3.5 + TypeScript 严格模式 + Pinia
- 11 个 .vue 组件、5 个 store、3 个 view、12 个 composable
- 总 Vue 代码约 3000 行（SmartInput / Timeline / SearchOverlay / FloatChat / QuickCapture / ItemDrawer / KeyGate / 旧 AtlasView / CaptureView 等）
- 设计 token 系统在 `src/styles/tokens.css`
- 构建产物 gzip ~47 KB 入口

### 6.2 Three.js 集成对比

**Three.js 本身 framework-agnostic**。它只需要一个 canvas 元素 + JS 控制权。

**Vue 方案**：
- `npm i three @types/three`
- 在 `<canvas ref="canvas">` 上 onMounted 启动 scene
- 摄像机 / scene / animation loop 作为 ref 持有
- onBeforeUnmount 调 `renderer.dispose()` / `scene.traverse(disposeMeshes)`
- **不用 TresJS**（Vue 包装层生态弱、文档少、bundle 额外开销）
- 状态用 Pinia store（cosmos store 持有树/关联数据 + 当前 state）

**React 方案**：
- `npm i three @react-three/fiber @react-three/drei`
- 用 r3f 的声明式 JSX `<mesh>` `<points>` 描述场景
- 优点：声明式状态绑定方便、社区例子多
- **代价**：项目必须换 React → 3000 行 Vue 代码全部作废、所有现有组件重写

### 6.3 决策

**保留 Vue，原生 Three.js（不用 TresJS）**。

理由：
- Three.js 在 Vue 里跑得跟在 React 里一样顺，集成成本不是技术决定项
- 重写 3000 行 Vue 代码代价不可接受
- 原生 Three.js 比 r3f 性能略好（少一层抽象）
- 后续如果做 Tauri 壳，单一 Vite 工程更简单

依赖增量：
- `three` (~140 KB gzip) — 但 Atlas 是独立 chunk，懒加载
- `@types/three` — 仅 dev

## 8. 开放问题（留给评审）

1. **单归属 vs 多归属**：v0.1 限制一个 entity 只能挂一条 tree_edge。是否同意？多归属让球面布局复杂但更真实
2. **focus_session 触发**：每次进 Atlas 都自动开 session 还是用户显式"开始一段思考"才记？倾向自动
3. **reflection_prompt 持久化**：v0.1 不存。v0.2 是否要存以分析"哪些反思打动用户"？
4. **关联阈值**：默认 0.7 是否合适？是否暴露给用户调？
5. **球外/球内混合**：是否第一版只做球外（简单），等 v0.2 再加球内？
6. **三模式入口位置**：Capture / Atlas / 近况 — 顶部 3-tab？还是 Capture 顶部嵌入 Atlas + 近况两个图标？
7. **路径回放**：focus_session 录的路径未来怎么呈现？季度末"行星会合"动画？还是简单列表？
8. **AI 关联拟合频率**：实时（每次 capture 后触发）还是离线批处理（每晚一次）？影响后端架构
9. **Atlas 进入引导**：每次都给"建议焦点"还是用户可关闭？
