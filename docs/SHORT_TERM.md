# Short Term

这份文档只讲短期目标。

它不负责讲长期愿景，也不负责讲完整上下文；它只负责回答一个问题：

`Axiom 现在最应该做什么？`

## 当前阶段

当前阶段：`v0.1 alpha`

当前主线：

```text
接收输入 -> 持久化存储 -> 基础检索
```

## 当前已经有的东西

- `/add`
- `/recent`
- `/search`
- SQLite `items` 表
- `data/inbox` 文件落盘
- `scripts/backup_axiom.py` 手动备份脚本

## 当前最重要的风险

### 运行稳定性

- `5000` 端口检查时未监听
- 正式启动方式还没有在仓库中固化
- 暂未确认正式的 `systemd` 服务

### 数据一致性

- inbox 文件数量与数据库记录数不一致
- `/add` 现在是先写文件，再写数据库
- 数据库失败时还没有补偿逻辑

### API 边界

- `/recent` 还没有最大 `limit`
- `/search` 和 `/recent` 的边界还不统一
- 错误返回仍是纯文本

### 备份闭环

- 手动备份脚本已经落地
- 但还没有在 VPS 上验证固定执行
- 也还没有做恢复演练

## 短期优先级

第一优先级：

- 明确 receiver 的正式启动方式
- 提升写入可靠性
- 保证文件和数据库一致性可检查
- 在 VPS 上确认备份脚本可执行

第二优先级：

- 给 `/recent` 增加最大 `limit`
- 统一 `/recent` 和 `/search` 的分页边界
- 统一错误返回格式

第三优先级：

- 图片上传
- 多类型 item 支持

## 当前建议顺序

1. 确认 receiver 如何稳定启动
2. 修 `/add` 的可靠性问题
3. 在 VPS 上跑一次真实备份
4. 做一次恢复演练
5. 再整理 `/recent` 和 `/search`

## 当前明确不做

- 不切换数据库
- 不引入新框架
- 不做复杂 agent
- 不做向量检索
- 不做多服务化
